# 缺失功能探索工作流設計

## 目標

建立專案級 prompt workflow，讓代理在使用者想盤點「專案還缺什麼功能」時，能根據現有證據共同發想、比較並排序候選功能，而不是直接憑直覺列願望清單。

## 使用情境

- 使用者詢問專案還缺哪些功能、下一步該做什麼或希望共同發想 roadmap。
- 使用者只有模糊方向，需要從現況找出產品、使用者旅程或工程能力缺口。
- 已有候選功能，需要用一致標準縮小範圍。

不適用於已明確定義需求的實作、單一 bug 診斷或既定規格的排程。

## 考慮方案

1. **自由對談腦力激盪**：速度快，但容易重複既有 roadmap、忽略證據，也難以比較價值。
2. **直接產生 roadmap**：輸出具體，但會過早把未驗證想法變成承諾。
3. **證據導向漏斗（採用）**：先盤點現況與使用者旅程，再分流發想、排序與形成提案；成本稍高，但結果可追溯且可執行。

## 工作流

1. **建立現況快照**：讀取 README、roadmap、規格、近期提交、主要入口與測試；分清楚已完成、已規劃、刻意不做與未知。
2. **描繪核心旅程**：用目標使用者的觸發、輸入、處理、輸出、回饋與復原步驟檢查中斷點。
3. **多視角產生候選**：至少從使用者價值、信任安全、操作效率、整合擴充與維運可觀測性發想；每項候選必須連到一個證據或明確假設。
4. **合併與過濾**：刪除重複、已存在、違反非目標或只有技術手段而沒有使用者成果的項目。
5. **排序**：以影響力、信心、成本、風險四項低／中／高評估，優先選高影響、高信心、成本可控的項目。
6. **共同收斂**：呈現 5–7 個候選與推薦前三名，讓使用者可挑選、改寫或要求再展開；若使用者授權代理自行決定，直接選定首選。
7. **輸出功能簡報**：為首選提供問題、目標使用者、使用情境、最小範圍、非目標、驗收條件、風險、證據與下一步。

## 輸出契約

輸出依序包含「現況摘要」、「缺口地圖」、「候選功能比較」、「推薦前三名」與「首選功能簡報」。未知資訊必須標記為假設，不可偽裝成已驗證事實；除非使用者另有要求，不直接修改 roadmap、建立 issue 或開始實作。

## 錯誤與邊界處理

- 文件互相矛盾時，列出矛盾並降低相關候選的信心。
- 專案資料不足時，提供帶假設標記的暫定結果，並把補證據列為下一步。
- 候選皆屬既有 roadmap 時，改找旅程中的落地缺口、驗收缺口或未覆蓋族群，不換句話重複條目。
- 不把大型願景當作單一功能；拆成能獨立驗收的最小成果。

## 驗證

- prompt workflow 具有合法 frontmatter，且位於 `.github/prompts/`。
- 靜態契約檢查確認工作流包含現況盤點、旅程分析、多視角候選、排序、前三名與功能簡報。
- 自我審查確認沒有 `TODO`、`TBD`、矛盾或超出需求的自動寫入行為。
