# 通用 AI 開發 Harness 設計

## 目的

建立不依賴特定 AI 供應商或代理框架的專案內工作協定，使能力不一的 AI 工具都能依相同階段理解需求、限制變更、留下證據並接受機械式驗證。

## 成功條件

- 新 AI 只需讀取根目錄 `HARNESS.md`，即可找到規則、狀態格式與標準命令。
- 每項工作以一份 JSON 狀態檔記錄需求、假設、範圍、計畫、變更與驗證證據。
- 驗證器可拒絕未知階段、缺少必要欄位、跳過階段門檻或證據不足的完成宣告。
- harness 不呼叫任何 AI API，也不綁定 Codex、Claude、Gemini 或其他產品。
- 驗證可在 Docker Compose 容器內執行，不需啟動應用程式伺服器或付費服務。

## 方案比較

1. **只有 Markdown 提示詞**：最容易採用，但無法阻止低品質代理跳過測試或虛構完成狀態。
2. **工具中立協定、JSON 狀態與驗證器（採用）**：保留跨工具相容性，同時提供可機械檢查的品質門檻。
3. **中央代理服務與 AI SDK**：可高度自動化，但新增常駐服務、供應商耦合與維護成本，超出目前需求。

## 架構

- `HARNESS.md`：人與 AI 的單一入口，定義順序、不可違反的規則、階段門檻與命令。
- `.harness/schema.json`：狀態檔的公開資料契約，供外部工具產生或整合。
- `.harness/task.example.json`：可複製的最小合法工作範例。
- `scripts/harness.py`：只使用 Python 標準函式庫的 `init` 與 `check` 指令。
- `tests/test_harness.py`：覆蓋初始化、基本驗證與完成門檻。

## 工作階段

階段固定為 `intake → design → plan → implement → verify → review → done`，另允許任何非完成工作進入 `blocked`。`phase_history` 必須記錄連續轉換，解除阻塞時回到原階段。驗證器檢查公開 JSON Schema、階段歷史與目前階段之前應具備的累積證據，不替 AI 執行命令，也不相信未記錄的口頭宣告。

## 完成門檻

`done` 必須同時具備：明確成功條件、非空計畫、至少一個變更檔案、至少一筆成功驗證命令、審查結果為 `pass`、無未解決事項。純文件或分析工作若沒有程式變更，仍需把實際產物列在 `changes.files`。

## 錯誤處理

驗證器一次列出所有已知錯誤並回傳非零狀態碼；JSON 語法錯誤、根節點型別錯誤與檔案不存在使用清楚訊息。`init` 預設拒絕覆寫既有檔案。

## 非目標

- 不負責選擇 AI 模型、分派代理、管理 token 或呼叫外部服務。
- 不取代 `AGENTS.md`、RFC、GitHub Project 或 Pull Request 流程。
- 不自動執行狀態檔內的任意 shell 命令，以免引入命令注入風險。

## 測試策略

使用 `unittest` 在暫存目錄執行 CLI。測試先確認不完整狀態被拒絕，再確認最小合法 `done` 狀態通過；完整驗證透過 Docker Compose 的一次性 `app` 容器執行。
